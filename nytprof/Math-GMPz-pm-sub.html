494=head1 NAME
495
496 Math::GMPz - perl interface to the GMP library's integer (mpz) functions.
497
498=head1 DEPENDENCIES
499
500 This module needs the GMP C library - available from:
501 http://gmplib.org
502
503=head1 DESCRIPTION
504
505 A bignum module utilising the Gnu MP (GMP) library.
506 Basically this module simply wraps nearly all of
507 the integer functions provided by that library.
508 The documentation below extensively plagiarises
509 the documentation at http://gmplib.org.
510 See the Math::GMPz test suite for examples of
511 usage.
512
513=head1 SYNOPSIS
514
515 use Math::GMPz qw(:mpz :primes :supp);
516
517 my $string = 'fa9eeeeeeeeeeeeea1234dcbaef1';
518 my $base = 16;
519
520 # Create the Math::GMPz object
521 my $bn1 = Rmpz_init_set_str($string, $base);
522
523 # Create another Math::GMPz object that holds
524 # an initial value of zero, but has enough
525 # memory allocated to store a 131-bit number.

527 # doesn't matter - additional memory is allocated
528 # automatically to Math::GMPz objects as needed
529 # by the GMP library.
530 my $bn2 = Rmpz_init2(131);
531
532 # Create another Math::GMPz object initialised to 0.
533 my $bn3 = Rmpz_init();
534
535 # or use the new() function:
536 my $bn4 = Math::GMPz->new(12345);
537
538 # Perform some operations ... see 'FUNCTIONS' below.
539
540 .

542

544 print Rmpz_get_str($bn1, 8), "\n";
545

554=head1 MEMORY MANAGEMENT
555
556 Objects created with the Rmpz_init* functions have
557 been blessed into package Math::GMPz. They will
558 therefore be automatically cleaned up by the DESTROY()
559 function whenever they go out of scope.
560
561 For each Rmpz_init* function there is a corresponding function
562 called Rmpz_init*_nobless which returns an
563 unblessed object. If you create Math::GMPz objects
564 using the '_nobless' versions, it will then be up to you
565 to clean up the memory associated with these objects by
566 calling Rmpz_clear($op) for each object. Alternatively
567 such objects will be cleaned up when the script ends.
568 I don't know why you would want to create unblessed
569 objects. The point is that you can if you want to.
570
571=head1 FUNCTIONS
572
573 See the GMP documentation at http://gmplib.org.
574
575 These next 3 functions are demonstrated above:
576 $rop = Rmpz_init_set_str($str, $base); # 1 < $base < 37
577 $rop = Rmpz_init2($bits); # $bits > 0
578 $str = Rmpz_get_str($r, $base); # 1 < $base < 37
579
580 The following functions are simply wrappers around a GMP
581 function of the same name. eg. Rmpz_swap() is a wrapper around
582 mpz_swap().
583
584 "$rop", "$op1", "$op2", etc. are Math::GMPz objects - the
585 return values of one of the Rmpz_init* functions.
586 They are in fact references to GMP structures.
587 The "$rop" argument(s) contain the result(s) of the calculation
588 being done, the "$op" argument(s) being the input(s) into that
589 calculation.
590 Generally, $rop, $op1, $op2, etc. can be the same perl variable,
591 though usually they will be distinct perl variables
592 referencing distinct GMP structures.
593 Eg something like Rmpz_add($r1, $r1, $r1),
594 where $r1 *is* the same reference to the same GMP structure,
595 would add $r1 to itself and store the result in $r1. Think of it
596 as $r1 += $r1. Otoh, Rmpz_add($r1, $r2, $r3), where each of the
597 arguments is a different reference to a different GMP structure
598 would add $r2 to $r3 and store the result in $r1. Think of it as
599 $r1 = $r2 + $r3.
600 Mostly, the first argument is the argument that
601 stores the result and subsequent arguments provide the input values.
602 Exceptions to this can be found in some of the functions that
603 actually return a value, and, eg., the div_qr functions (which
604 yield both quotient and remainder as their first *two* arguments).
605 Like I say, see the GMP manual for details. I hope it's
606 intuitively obvious or quickly becomes so. Also see the test
607 suite that comes with the distro for some examples of usage.
608
609 "$ui" means any integer that will fit into a C 'unsigned long int'.
610
611 "$si" means any integer that will fit into a C 'signed long int'.
612
613 "$double" means any number (not necessarily integer) that will fit
614 into a C 'double'.
615
616 "$bool" means a value (usually a 'signed long int') in which
617 the only interest is whether it evaluates as true or not.
618
619 "$str" simply means a string of symbols that represent a number,
620 eg "1234567890987654321234567" which might be a base 10 number,
621 or "zsa34760sdfgq123r5" which would have to represent a base 36
622 number (because "z" is a valid digit only in base 36). Valid
623 bases for GMP numbers are 2 to 62 (inclusive).

1249 OPERATOR OVERLOADING
1250
1251 The following operators are overloaded:
1252 + - * / %
1253 += -= *= /= %=
1254 << >> <<= >>=
1255 & | ^ ~
1256 &= |= ^=
1257 < <= > >= == != <=>
1258 ! not abs
1259
1260 Division uses 'tdiv' (see 'Integer Division', above).
1261 Check that '~', '%', and '%=' are working as you expect
1262 (especially in relation to negative values).
1263
1264 In those situations where the overload subroutine operates on 2
1265 perl variables, then obviously one of those perl variables is
1266 a Math::GMPz object. To determine the value of the other variable
1267 the subroutine works through the following steps (in order),
1268 using the first value it finds, or croaking if it gets
1269 to step 6:
1270
1271 1. If the variable is an unsigned long then that value is used.
1272 The variable is considered to be an unsigned long if
1273 (perl 5.8) the UOK flag is set or if (perl 5.6) SvIsUV()
1274 returns true.
1275
1276 2. If the variable is a signed long int, then that value is used.
1277 The variable is considered to be a signed long int if the
1278 IOK flag is set. (In the case of perls built with
1279 -Duse64bitint, the variable is treated as a signed long long
1280 int if the IOK flag is set.)
1281
1282 3. If the variable is a double, then that value is used. The
1283 variable is considered to be a double if the NOK flag is set.
1284
1285 4. If the variable is a string (ie the POK flag is set) then the
1286 value of that string is used. Octal strings must begin with
1287 '0', hex strings must begin with either '0x' or '0X' -
1288 otherwise the string is assumed to be decimal. If the POK
1289 flag is set, but the string is not a valid base 8, 10, or 16
1290 number, the subroutine croaks with an appropriate error
1291 message.
1292
1293 5. If the variable is a Math::GMPz object then the value of that
1294 object is used.
1295
1296 6. If none of the above is true, then the second variable is
1297 deemed to be of an invalid type. The subroutine croaks with
1298 an appropriate error message.
1299
1300 If the second operand is a 'double' (ie if the other
1301 operand's NOK flag is set) then it is first truncated
1302 to an integer value before the operation is performed.
1303
1304 For example:
1305 my $x = Rmpz_init_set_ui(112);
1306 $x *= 2.9;
1307 print "$x"; # prints 224
1308
1309 Atempting to use the overloaded operators with objects that
1310 have been blessed into some package other than 'Math::GMPz'
1311 will not work.
